🔒 PERMISSION & SECURITY ALIGNMENT
═══════════════════════════════════════════════════════════════════════════════

FIRESTORE RULES LOGIC:
1. User must be authenticated
2. User must be the initiator (creating call)
3. User must be a member of the group (verified via dmChats/{groupId}.members)
4. User must be in the participants list
5. Participants list must be 2-6 members
6. All required fields must be valid types (string, list, timestamp)
7. Only participants can read/update the call

PERMISSION FLOW:
┌─────────────────────────────────────────────────────────────────────────┐
│ 1. User taps call button                                                │
│ 2. canStartGroupCall() checks:                                          │
│    → User is group member                                               │
│    → Group settings allow calls (or user is admin)                      │
│ 3. getActiveGroupMembers() fetches all members                          │
│ 4. startGroupAudioCall() creates call with:                             │
│    → groupId                                                             │
│    → initiatorId (current user)                                         │
│    → participants (all group members)                                   │
│ 5. Firestore rules validate:                                            │
│    → User is initiator ✓                                                │
│    → User is in participants ✓                                          │
│    → User is group member (via dmChats read) ✓                          │
│    → Data structure is valid ✓                                          │
│ 6. Call document created successfully                                   │
└─────────────────────────────────────────────────────────────────────────┘

NO PERMISSION ERRORS EXPECTED - All operations are aligned!
